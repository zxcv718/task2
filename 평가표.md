## 기능 동작 검증

### 1. 프로그램 실행 시 메뉴가 정상적으로 표시되고, 퀴즈 풀기/추가/목록/점수 확인 기능이 모두 동작하는가?

**답변**  
예. `main.py`는 `QuizGame()` 객체를 생성한 뒤 `run()`을 호출하고, `QuizGame.run()`은 메뉴를 반복 출력하면서 입력값에 따라 기능을 분기합니다.

현재 메뉴는 다음 6개입니다.

- `1. 퀴즈 풀기`
- `2. 퀴즈 추가`
- `3. 퀴즈 목록`
- `4. 퀴즈 삭제`
- `5. 점수 확인`
- `6. 종료`

즉 평가표 필수 기능인 퀴즈 풀기, 퀴즈 추가, 퀴즈 목록, 점수 확인, 종료가 모두 구현되어 있고, 보너스 기능인 퀴즈 삭제도 함께 들어 있습니다. 메뉴 분기는 `QuizGame`이 맡고, 실제 세부 동작은 `QuizSessionRunner`, `QuizCatalogManager`, `scoreboard`, `StateStore` 같은 하위 모듈에 위임하는 구조입니다.

### 2. 퀴즈 풀기 시 정답/오답 판정이 정확하고, 최소 입력 오류 케이스(공백/문자/범위 밖/빈 입력)에 대한 처리가 되어 있는가?

**답변**  
예. 정답 판정은 `quiz.py`의 `Quiz.is_correct()`가 담당하며, 사용자가 입력한 번호와 정답 번호를 직접 비교합니다.

입력 검증은 단계별로 나뉘어 있습니다.

- `console_io.prompt_text()`가 앞뒤 공백 제거와 빈 입력 방지를 처리합니다.
- `console_io.prompt_int()`가 숫자 변환 실패와 범위 검사를 처리합니다.
- `quiz_session._prompt_answer_for_quiz()`가 퀴즈 풀이 중 `1`~`4` 또는 `h`만 허용합니다.

따라서 공백 입력, 빈 입력, 문자 입력, 범위 밖 숫자 입력에 대한 처리가 모두 들어 있습니다. 테스트에서도 잘못된 입력 뒤에 올바른 답을 다시 받았을 때 `answered_count`가 유효한 답 입력 기준으로만 증가하는 흐름이 검증됩니다.

### 3. 프로그램 종료 후 재실행해도 추가한 퀴즈와 최고 점수가 유지되는가?

**답변**  
예. 현재 상태는 프로젝트 루트의 `state.json`에 저장됩니다. `QuizGame.__init__()`는 시작할 때 `StateStore.load_state()`를 호출해 저장 파일을 읽고, 퀴즈 추가, 퀴즈 삭제, 퀴즈 플레이 종료, 프로그램 종료 시점마다 `save_state()`를 호출합니다.

저장되는 핵심 값은 `quizzes`, `next_quiz_id`, `best_score`, `history`입니다. 그래서 프로그램을 다시 실행하면 추가한 퀴즈, 최고 점수, 플레이 기록이 그대로 복원됩니다. 파일이 없으면 기본 퀴즈 데이터로 새 `state.json`을 만들고, 파일이 손상되었거나 스키마가 잘못되었으면 안내 메시지를 출력한 뒤 기본 데이터로 복구합니다.

## 코드 구조 및 설계

### 8. `Quiz`와 `QuizGame` 등 클래스들의 책임을 어떻게 나눴는지 설명할 수 있는가?

**답변**  
예. 현재 구조는 최소 2개 클래스 조건을 넘어서, 역할별 클래스를 여러 개로 나눈 구조입니다.

- `Quiz`
  퀴즈 1문제를 표현하는 데이터 모델입니다. `quiz_id`, `question`, `choices`, `answer`, `hint`를 보관하고, 생성 시 유효성 검사와 정답 확인, 출력 문자열 생성, 딕셔너리 변환을 담당합니다.
- `QuizGame`
  프로그램 전체 흐름을 조율하는 상위 객체입니다. 메뉴 출력, 메뉴 분기, 저장 시점 결정, 안전 종료를 담당합니다.
- `QuizSessionRunner`
  한 번의 플레이 세션을 담당합니다. 문제 수 선택, 랜덤 출제, 힌트 처리, 채점, 중단 시 부분 결과 계산을 맡습니다.
- `QuizCatalogManager`
  퀴즈 추가, 목록 출력, 삭제를 담당합니다.
- `StateStore`
  `state.json` 읽기/쓰기, 스키마 검증, 복구를 담당합니다.
- `GameState`
  저장 파일에서 읽어 온 상태를 메모리에서 다루기 쉽게 묶는 데이터 구조입니다.
- `HistoryEntry`
  플레이 히스토리 1건의 생성, 검증, 직렬화를 담당하는 데이터 모델입니다.

즉 현재 구조는 "문제 모델", "게임 전체 제어", "플레이 세션", "퀴즈 관리", "저장소 계층", "히스토리 모델"로 책임이 나뉘어 있습니다.

### 9. “입력 처리(검증)”, “게임 진행”, “데이터 저장/불러오기” 로직을 어떤 기준으로 분리했는지 설명할 수 있는가?

**답변**  
현재 코드는 이 세 영역을 파일과 클래스 단위로 나누었습니다.

- 입력 처리와 검증: `console_io.py`
  `prompt_text()`, `prompt_int()`, `SafeExitRequest`를 통해 공통 입력 정책을 한 곳에 모읍니다.
- 게임 진행: `quiz_game.py`, `quiz_session.py`
  `QuizGame`이 메뉴와 전체 흐름을 관리하고, `QuizSessionRunner`가 한 번의 플레이 세부 진행을 담당합니다.
- 데이터 저장/불러오기: `state_store.py`
  `StateStore.load_state()`, `save_state()`, `_parse_state()`가 파일 입출력과 최상위 state 검증을 담당합니다.
- 히스토리 스키마/직렬화: `history_entry.py`
  `HistoryEntry.create_now()`, `from_dict()`, `to_dict()`가 history 한 건의 생성과 검증, JSON 변환을 담당합니다.
- 퀴즈 관리: `quiz_catalog.py`
  퀴즈 추가, 목록 출력, 삭제를 별도 모듈로 분리했습니다.
- 점수/히스토리 표시: `scoreboard.py`
  이미 만들어진 `HistoryEntry` 목록을 읽기 좋게 출력하는 역할만 담당합니다.

이렇게 나누면 입력 정책, 한 판의 진행 방식, 저장 포맷, history 스키마, 출력 규칙이 서로 덜 섞여서 수정 범위를 더 명확하게 잡을 수 있습니다.

### 10. `state.json` 읽기/쓰기 흐름이 프로그램의 어디에서, 어떤 순서로 발생하는지 설명할 수 있는가?

**답변**  
예. 읽기 흐름은 다음과 같습니다.

1. `main.py`에서 `QuizGame()`을 생성합니다.
2. `QuizGame.__init__()`가 `load_state()`를 호출합니다.
3. `QuizGame.load_state()`가 `StateStore.load_state()`를 호출합니다.
4. `StateStore.load_state()`는 `state.json` 존재 여부를 확인합니다.
5. 파일이 있으면 `json.load()`로 JSON을 읽고 `_parse_state()`로 검증합니다.
6. `_parse_state()`는 최상위 키와 값 범위, `quizzes`, `history` 구조를 검사합니다.
7. `quizzes`는 `Quiz.from_dict()`로 복원되고, `history`는 `HistoryEntry.from_dict()`로 복원됩니다.
8. 검증이 끝나면 `GameState` 객체로 묶어 반환하고, `QuizGame`이 이를 자신의 속성에 반영합니다.
9. 파일이 없거나 JSON이 손상되었거나 필수 키가 빠진 경우에는 기본 상태를 만들고 새 파일을 저장합니다.

쓰기 흐름은 다음과 같습니다.

1. `QuizGame.save_state()`가 현재 메모리 상태를 `GameState`로 묶습니다.
2. `StateStore.save_state()`가 이를 JSON 직렬화 가능한 딕셔너리로 변환합니다.
3. 각 퀴즈는 `Quiz.to_dict()`로 변환됩니다.
4. 각 history 항목은 `HistoryEntry.to_dict()`로 변환됩니다.
5. 마지막에 `json.dump(..., ensure_ascii=False, indent=2)`로 UTF-8 `state.json`을 씁니다.

즉 시작할 때는 `파일 -> 검증 -> 메모리`, 저장할 때는 `메모리 -> 직렬화 -> 파일` 순서입니다.

### 11. `Ctrl+C` 또는 EOF 상황에서 “안전 종료”를 위해 어떤 처리를 했는지(또는 어떤 처리가 필요했는지) 설명할 수 있는가?

**답변**  
현재 구현은 입력을 받는 가장 안쪽 지점인 `prompt_text()`에서 `KeyboardInterrupt`와 `EOFError`를 잡고, 이를 `SafeExitRequest`로 변환합니다. 이 예외를 일반 입력 오류와 구분해서 "저장 후 종료" 흐름으로 연결한 것이 핵심입니다.

메뉴 입력 단계에서 중단되면 다음 순서로 동작합니다.

1. `prompt_text()`가 중단 예외를 감지합니다.
2. 안내 메시지를 출력합니다.
3. `SafeExitRequest`를 발생시킵니다.
4. `QuizGame.run()`이 이를 받아 `safe_exit()`를 호출합니다.
5. `safe_exit()`가 가능한 범위에서 현재 상태를 저장하고 종료 메시지를 출력합니다.

퀴즈 풀이 도중 중단되면, 세션 내부에서 지금까지의 결과를 `QuizSessionResult(interrupted=True)`로 반환하고, 상위 `QuizGame.play_quiz()`가 현재까지의 결과를 `HistoryEntry`로 기록한 뒤 저장하고 종료 흐름으로 넘깁니다. 그래서 비정상 종료처럼 끝나지 않고, 부분 결과도 잃지 않게 설계되어 있습니다.

### 12. Git 커밋을 어떤 단위로 나누었고, 커밋 메시지 규칙을 어떻게 정했는지 설명할 수 있는가?

**답변**  
현재 커밋은 대체로 기능 단위와 리팩터링 단위로 나뉘어 있습니다. 처음에는 `Quiz` 모델, 기본 데이터, 진입점, 통합 게임 흐름처럼 큰 기능을 올리고, 이후에는 상수 분리, 입력 처리 분리, 저장소 분리, 퀴즈 진행 분리, 점수판 분리, `HistoryEntry` 도입 같은 책임 분리 리팩터링을 커밋으로 나눴습니다.

메시지 규칙은 변경 성격을 접두어로 드러내는 방식입니다.

- `Feat`: 기능 추가
- `Refactor` 또는 `refactor`: 구조 개선
- `Fix`: 버그 수정
- `Docs` 또는 `docs`: 문서 변경
- `Chore`: 설정/정리 작업

최근 예시는 다음과 같습니다.

- `refactor: introduce HistoryEntry and simplify module boundaries`
- `fix: normalize quiz ids and improve scoreboard history display`
- `docs: update README for history ordering and scoreboard layout`
- `docs: refresh evaluation sheet for current implementation`

즉 커밋 단위는 "무엇을 바꿨는지 한 문장으로 설명 가능한 정도"로 나눈 편입니다.

## 핵심 기술 원리 적용

### 13. 클래스를 사용한 이유는 무엇이며, 함수만으로 구현할 때와 어떤 차이가 있는지 설명할 수 있는가?

**답변**  
이 프로젝트에서 클래스를 쓴 가장 큰 이유는 "관련된 데이터와 동작을 함께 묶기 위해서"입니다.

예를 들어 `Quiz`는 문제 문장, 선택지, 정답, 힌트 같은 데이터를 들고 있으면서 동시에 정답 확인과 출력 문자열 생성도 담당합니다. `HistoryEntry`도 마찬가지로, 히스토리 데이터 자체와 그 검증/직렬화 규칙을 함께 보유합니다.

만약 함수만으로 구현하면 퀴즈 데이터와 history 데이터가 여러 함수에 흩어지고, 매번 필요한 필드를 따로 전달해야 해서 실수가 늘어날 수 있습니다. 현재 구조에서는 `Quiz`, `HistoryEntry`, `QuizGame`, `QuizSessionRunner`, `QuizCatalogManager`, `StateStore`가 각자 자기 상태와 책임을 들고 있기 때문에, 메뉴 제어, 플레이 진행, 저장 처리, 퀴즈 관리, history 스키마가 서로 덜 섞입니다.

즉 클래스는 이 프로젝트에서 단순히 문법적 장식이 아니라, 역할 분리와 상태 관리 구조를 명확하게 만드는 도구입니다.

### 14. JSON 파일로 데이터를 저장하는 이유와 JSON 형식의 특징을 설명할 수 있는가?

**답변**  
JSON을 사용하는 이유는 현재 저장해야 하는 데이터 구조가 Python의 `dict`, `list`, 문자열, 숫자와 잘 맞기 때문입니다. 이 프로젝트의 상태는 `version`, `next_quiz_id`, `best_score`, `quizzes`, `history` 같은 키를 가진 딕셔너리 구조이고, JSON이 이런 형태를 그대로 표현하기 좋습니다.

현재 저장 구조는 크게 두 덩어리입니다.

- `quizzes`
  각 퀴즈를 `id`, `question`, `choices`, `answer`, `hint`로 저장합니다.
- `history`
  각 플레이 기록을 `played_at`, `selected_count`, `answered_count`, `correct_count`, `score`, `hint_used_count`로 저장합니다.

메모리 안에서는 `history`를 `HistoryEntry` 객체로 다루지만, 파일에 저장할 때는 `HistoryEntry.to_dict()`를 통해 JSON-friendly 구조로 바꿉니다. 이 방식은 메모리 모델을 명확하게 유지하면서도 파일 포맷은 단순하게 유지할 수 있다는 장점이 있습니다.

### 15. 파일 입출력에서 `try/except`가 필요한 이유(발생 가능한 실패 케이스 포함)를 설명할 수 있는가?

**답변**  
파일 입출력은 코드가 맞더라도 외부 환경 때문에 실패할 수 있으므로 `try/except`가 필요합니다. 특히 이 프로젝트는 `state.json`이 프로그램 실행의 시작점과 끝점 모두에 관여하므로, 읽기나 쓰기 실패를 그냥 방치하면 프로그램 전체가 쉽게 멈출 수 있습니다.

현재 코드가 실제로 대비하는 실패 케이스는 다음과 같습니다.

- 파일이 존재하지 않음
- JSON 문법이 깨져 있음
- 필수 키가 빠져 있음
- 값 타입이 예상과 다름
- 값 범위가 잘못됨
- 디스크/권한 문제 등으로 파일 읽기나 쓰기가 실패함

현재 구현에서 `StateStore.load_state()`는 `OSError`, `json.JSONDecodeError`, `KeyError`, `TypeError`, `ValueError`를 잡아 기본 데이터로 복구하고, `save_state()`는 `OSError`를 잡아 저장 실패 메시지를 출력합니다. 즉 `try/except`는 예외를 숨기기 위한 장치가 아니라, 실패했을 때도 프로그램이 다시 실행 가능하도록 만드는 안정성 장치입니다.

### 16. 브랜치를 분리해 작업한 이유와 병합(`merge`)의 의미를 설명할 수 있는가?

**답변**  
브랜치를 분리하는 이유는 아직 완성되지 않은 기능이나 리팩터링 작업을 `main`과 분리해 진행하기 위해서입니다. 이 프로젝트에서는 `feature/split-quiz-game` 브랜치에서 `quiz_game.py`의 책임을 여러 파일로 나누는 구조 개선 작업을 진행했고, 완료 후 `main`으로 병합했습니다.

병합의 의미는 별도 작업선에서 만든 결과를 다시 메인 흐름에 합치는 것입니다. 현재 저장소에서는 `8ce280f Merge: feature/split-quiz-game 브랜치 병합` 커밋이 그 역할을 합니다. 즉 실제 Git 로그에서도 "분리해서 작업하고, 검토 가능한 단위로 합쳤다"는 흐름이 드러납니다.

### 17. `state.json` 데이터 구조(필드/중첩 구조)를 현재 형태로 설계한 이유를 설명할 수 있는가?

**답변**  
현재 `state.json`은 "게임 전체 상태를 한 파일에서 읽고 쓰기 쉬운 구조"로 설계되어 있습니다. 최상위 필드가 각각 한 가지 역할을 가지기 때문에, 저장과 복원이 단순해집니다.

최상위 필드는 다음 의미를 가집니다.

- `version`
  현재 코드가 이해할 수 있는 저장 스키마 버전인지 확인하기 위한 값입니다.
- `next_quiz_id`
  현재 등록된 마지막 퀴즈 번호 다음 값을 가리켜, 삭제 후 번호를 다시 매긴 뒤에도 새 퀴즈를 연속 번호로 추가하기 위한 값입니다.
- `best_score`
  최고 점수를 빠르게 출력하기 위한 값입니다.
- `quizzes`
  현재 등록된 모든 문제 목록입니다.
- `history`
  플레이할 때마다 쌓이는 기록 목록입니다. 점수 확인 화면에서는 오래된 기록부터 최신 기록까지 시간순으로 보여 주며, 가장 최근 기록이 맨 마지막에 옵니다.

`history`를 파일에서는 dict 리스트로 두고, 메모리에서는 `HistoryEntry` 객체 리스트로 다루는 이유도 분명합니다. 파일은 단순한 JSON 구조가 읽기 쉽고 저장이 편하고, 메모리에서는 전용 모델을 써야 검증과 출력 규칙이 한 곳에 모입니다.

특히 `selected_count`와 `answered_count`를 분리한 점이 중요합니다. 현재 코드는 유효한 최종 답안이 입력된 문제만 `answered_count`에 반영하므로, 잘못된 입력이나 힌트 요청이 반복되더라도 실제로 푼 문제 수가 부풀려지지 않습니다.

## 심층 인터뷰

### 18. 퀴즈 데이터가 1000개 이상으로 늘어난다면 현재 JSON 저장 방식에 어떤 한계가 생길 수 있는지 설명할 수 있는가?

**답변**  
현재 방식은 `state.json` 전체를 한 번에 읽고, 저장할 때도 전체를 다시 씁니다. 그래서 퀴즈가 1000개 이상으로 커지면 다음 한계가 생길 수 있습니다.

- 프로그램 시작 시 로드 시간이 길어질 수 있습니다.
- 퀴즈 하나만 바꿔도 전체 파일을 다시 저장해야 합니다.
- 파일 크기가 커져 메모리 사용량과 입출력 비용이 함께 늘어납니다.
- 사람이 직접 파일을 열어 확인하거나 충돌을 해결하기가 점점 어려워집니다.
- strict validation도 전체 데이터에 대해 수행되므로 초기 로드 부담이 커집니다.

즉 현재 방식은 교육용 콘솔 프로젝트 규모에는 적합하지만, 데이터가 크게 늘어나면 SQLite 같은 데이터베이스나 더 세분화된 저장 방식이 더 자연스럽습니다.

### 19. 만약 `state.json`이 손상되어 JSON 파싱에 실패한다면, 사용자가 데이터를 잃지 않도록 어떤 대응(복구/백업/초기화)이 가능한지 설명할 수 있는가?

**답변**  
현재 구현은 JSON 파싱에 실패하거나 스키마 검증에 실패하면 안내 메시지를 출력하고, 기본 상태를 다시 만들어 새 `state.json`을 저장합니다. 장점은 프로그램이 바로 다시 실행 가능해진다는 점입니다.

다만 데이터 보존 관점에서는 보완 여지가 있습니다. 더 안전한 대응은 다음과 같습니다.

1. 손상된 원본을 바로 덮어쓰지 않고 `state.json.bak`처럼 백업합니다.
2. 기본 상태로 새 `state.json`을 생성합니다.
3. 사용자에게 기존 파일을 백업했다는 메시지를 보여 줍니다.
4. 저장할 때는 임시 파일에 먼저 쓴 뒤, 성공했을 때만 원본과 교체하는 방식으로 바꿉니다.

즉 현재 코드는 "즉시 실행 가능하게 복구"하는 방식이고, 더 발전시키려면 "원본 보존 + 새 파일 생성" 전략을 추가하는 것이 좋습니다.

### 20. “정답 채점 방식(점수 계산)”이나 “퀴즈 구조(선택지 개수 등)” 요구사항이 바뀐다면, 어떤 파일/클래스/메서드를 먼저 수정해야 하는지 설명할 수 있는가?

**답변**  
점수 계산 규칙이 바뀐다면 가장 먼저 봐야 할 곳은 `quiz_session.py`와 `game_constants.py`입니다.

- `game_constants.py`
  현재 기본 점수 `BASE_POINTS`, 힌트 사용 점수 `HINT_POINTS`가 정의되어 있습니다.
- `quiz_session.calculate_points()`
  정답 여부와 힌트 사용 여부에 따라 실제 획득 점수를 계산합니다.
- `quiz_game.py`
  결과 요약과 `best_score` 갱신 흐름을 함께 봐야 합니다.
- `scoreboard.py`
  점수 출력 형식이나 라벨이 바뀌면 이 파일도 함께 봐야 합니다.

퀴즈 구조가 바뀐다면 수정 범위는 더 넓습니다.

- `quiz.py`
  `Quiz.__post_init__()`가 선택지 개수와 정답 범위를 검사하므로 가장 먼저 수정해야 합니다.
- `quiz.py`
  `display()`, `to_dict()`, `from_dict()`도 새 구조에 맞춰 바뀌어야 합니다.
- `quiz_catalog.py`
  현재는 선택지 4개를 고정 입력받으므로 입력 흐름을 바꿔야 합니다.
- `quiz_session.py`
  `_prompt_answer_for_quiz()`의 허용 답안 범위도 함께 바뀌어야 합니다.
- `default_data.py`
  기본 퀴즈 seed 형식도 새 구조에 맞춰 수정해야 합니다.
- `state_store.py`
  저장/복원 스키마 검증이 달라지면 여기 역시 같이 수정해야 합니다.

history 구조가 바뀐다면 `history_entry.py`를 가장 먼저 봐야 하고, 그다음 `state_store.py`, `quiz_game.py`, `scoreboard.py`를 같이 수정하는 것이 가장 자연스럽습니다.
